# Принципы unit-тестирования в проекте

## 1. Тест — это документация поведения

Имя теста должно читаться как предложение, описывающее **что** делает код в **каком** сценарии. Реализация (вызовы методов, моки) — детали.

```kotlin
// ✅ Хорошо — описано поведение
fun `should return failure when repository throws network error`()

// ❌ Плохо — описана реализация
fun `testValidatePhoneNumber1`()
fun `repository_validatePhone_called_once`()
```

## 2. Один тест — одно поведение

Если в `@Test` больше одного `assert` на разные аспекты — это два теста. Сплитни.

Исключение: несколько ассертов на разные поля одного результата (один объект → один тест).

## 3. AAA / Given-When-Then

Каждый тест разделён на 3 фазы — визуально (пустые строки или комментарии).

```kotlin
@Test
fun `should map phone with spaces when toUi called`() {
    // Given
    val state = State(phone = "+7 900 123 45 67")

    // When
    val uiState = state.toUi()

    // Then
    assertEquals("+7 900 123 45 67", uiState.phone)
}
```

## 4. Тестируй публичный контракт, а не реализацию

- ✅ Проверяй то, что наблюдаемо снаружи: возвращаемое значение, изменения State, опубликованные Event, вызовы зависимостей.
- ❌ Не проверяй приватные методы, внутренние поля.

Если приватный метод требует тестирования — его пора выделить в отдельный класс / use case.

## 5. F.I.R.S.T

| Свойство | Что значит |
|----------|------------|
| **F**ast | Тест работает миллисекунды. Никаких реальных I/O, sleep, сетей, файловой системы. |
| **I**ndependent | Тесты не зависят от порядка выполнения. Никаких `companion object var`-флагов между тестами. |
| **R**epeatable | Запустил 100 раз — 100 одинаковых результатов. Никаких `Random`, `LocalDateTime.now()` без инжекта Clock. |
| **S**elf-validating | Тест либо зелёный, либо красный. Никаких `println` для ручной проверки. |
| **T**imely | Пишутся вместе с production-кодом, а не "потом". |

## 6. Минимум 2 теста на единицу поведения

- **Happy path** — нормальное выполнение
- **Failure path** — ошибка зависимости / невалидный вход / граничный случай

## 7. Edge cases — обязательно

Прежде чем закончить, прокатай в голове:
- Пустая строка / пустой список
- `null` (если поле nullable)
- 0 / отрицательное число / `Int.MAX_VALUE`
- Длинная строка / спецсимволы / эмодзи / кириллица
- Дублирующиеся вызовы (idempotency)
- Параллельные вызовы (если применимо)

## 8. Не тестируй фреймворки

- Не тестируй, что Kotlin `data class` правильно делает `copy()`.
- Не тестируй, что `kotlinx.serialization` правильно сериализует `@Serializable`.
- Не тестируй, что Compose `MutableState` обновляется.
- Не тестируй, что MviKotlin Store сам по себе работает (это библиотечный контракт).

Тестируй **свою** логику.

## 9. Стабильность важнее покрытия

Лучше 60% стабильно зелёного покрытия, чем 90% с регулярными flaky-падениями. Flaky тест → удали или почини в тот же день.

## 10. Тест — это код

К тестам применяются те же правила качества: имена, структура, отсутствие дублирования (выноси `setup()`, фабрики тестовых данных в `Factory.kt` файлы).

```kotlin
// src/test/java/.../testdata/AuthTestData.kt
object AuthTestData {
    fun validationDto(
        phone: String = "+79001234567",
        password: String = "password123"
    ) = ValidationPhoneDto(phone, password)
}
```
