# Сохранения текущего состояния выполнения задачи
Этот skill позволяет сохранить текущий контекст выполнения задачи в промежуточный файл. Далее этот файл можно будет использовать
как базовый контекст для новой сессии Claude. В этой сессии можно будет продолжить выполнение текущих задач.

## Общие правила заполнения

- **Цель файла** — дать новой сессии Claude всё, чтобы она продолжила работу без повторного исследования. Не дублируй содержимое плана (если есть отдельный plan-файл) — ссылайся на него по абсолютному пути.
- **Абсолютные даты** — все даты записывай в формате `YYYY-MM-DD`. Никаких "сегодня", "вчера", "на этой неделе".
- **Не записывай секреты** — токены, пароли, приватные ключи, содержимое `.env`. Если они часть контекста — упомяни сам факт ("используется API-ключ из переменной окружения `X`"), но не значение.
- **Перед сохранением проверь, существует ли файл** — если `STATE_COMPRESSION.md` уже есть, прочитай его и **обнови** релевантные разделы, а не перезаписывай слепо. Прежняя информация о решениях/договорённостях обычно ценна.
- **Пиши в третьем лице / инфинитиве** — не "я сделал X", а "X реализован" или "реализовать X". Файл читает другая сессия Claude.
- **Точные пути файлов** — всегда абсолютные или от корня проекта. Без относительных путей вида `../../foo`.
- **Конкретика > абстракция** — не "разобраться с навигацией", а "проверить, что `Config.ForgotPassword` уже зарегистрирован в `RootAuthComponentImpl.kt:50`".

## Структура файла

Файл — markdown со следующими разделами (порядок важен — от общего к частному):
- [ ] Цель
- [ ] Текущий план
- [ ] Текущие факты
- [ ] Релевантные файлы
- [ ] Текущие решения
- [ ] Риски
- [ ] Следующие шаги

### Цель
Что писать:
- Что именно реализуется (1-2 предложения, без воды).
- **Scope** — что входит в задачу.
- **Out of scope** — что явно НЕ входит (часто именно это спасает следующую сессию от ненужной работы).

**Пример:**
> Реализовать экран **ForgotPassword**: поле email + кнопка "Отправить", regex-валидация на потерю фокуса, вызов `ForgotPasswordUseCase` из MviKotlin executor'а, `pop()` по успеху.
>
> **Out of scope:** добавление ссылки "Забыли пароль?" в `PhoneValidationScreen` — точка входа в этот экран будет подключена отдельной задачей.

### Текущий план
Описывает текущий план реализации фичи — пронумерованный список задач, которые нужно выполнить.

Перед каждым пунктом ставится ✅ если задача выполнена, ⏳ если ожидает выполнения. У каждого пункта — конкретный артефакт (файл, метод, команда), а не абстрактное действие.

**Пример описания текущего плана:**
1. ✅ **Регистрация модулей** — `settings.gradle.kts`: добавлены `include(":products-api")`, `include(":products-impl")`.
2. ✅ **`products-api`** — `RootProductsComponent`, `ProductListComponent`, `ProductDetailComponent` (interface + UiState + Callbacks + Factory) + `build.gradle.kts`.
3. ✅ **Component + Screen detail** — `ProductDetailComponentImpl`, `ProductDetailScreen` (TopAppBar + контент).
4. ⏳ **Root products** — `RootProductsComponentImpl` со `StackNavigation<Config>`, `Config.List`/`Config.Detail(productId)`.
5. ⏳ **DI** — `ProductsGraph` (`@DependencyGraph` + `@Binds` фабрик и use-case'ов + `@DependencyGraph.Factory` принимающий `TeaFactory`).

### Текущие факты
Описывает важные факты, которые могут понадобиться Claude Code, чтобы корректно понять задачу в новой, чистой сессии. Включает то, что уже невозможно вывести просто чтением кода.

Что писать:
- **Что уже сделано / в каком состоянии находится работа** (закоммичено? в staged? только в плане?).
- **Договорённости с пользователем** — ответы на `AskUserQuestion` или явные указания в чате (например "точку входа НЕ добавлять — только сам экран").
- **Релевантные memory notes** из `CLAUDE.md` / `MEMORY.md`, которые касаются именно этой задачи (например: "Metro DI не поддерживает несколько `@Assisted` параметров одного типа").
- **Внешние зависимости** — что готово в смежных слоях (API/Domain/Data), что замокано, что ещё не реализовано.
- **Тонкие места проекта** — версии тулов, gradle-таски с особенностями (`./gradlew test` запускает unit-тесты всех модулей; для конкретного модуля — `:<module>:testDebugUnitTest`).

Формат: маркированный список или подразделы. Каждый факт — отдельная строка/абзац.

### Релевантные файлы
Перечень файлов, с которыми взаимодействует задача. Для каждого — путь + роль.

Группировать по категориям:
- **Уже существуют (использовать как есть, не изменять)** — DI, навигация, API-контракты.
- **Файлы к созданию** — список с полными путями.
- **Файлы к изменению** — что и зачем.
- **Образцы для копирования паттернов** — какой существующий компонент брать за основу (важно: следующая сессия часто экономит время, увидев "копируй с `RegistrationComponentImpl.kt`").

Формат — таблица `| Файл | Роль |` или список с `—`. Только абсолютные пути или пути от корня проекта.

### Текущие решения
Уже принятые архитектурные/технические решения **с обоснованием**. Без "почему" решение легко пересмотреть в следующей сессии без причины и сломать договорённости.

Что писать:
- Архитектурные выборы (где живёт логика — в reducer как чистая функция или в executor как side effect; способ обработки ошибок).
- Выбор библиотек/API.
- Договорённости по scope (например: "точка входа НЕ реализуется — выбор пользователя").
- Решения по конкретным API: regex-формула, dispatchers, способ обработки ошибок.

Формат: пронумерованный список. Каждый пункт: **что решено** — **почему / откуда взялось решение**.

**Пример:**
1. **Точка входа НЕ реализуется в этой задаче** — выбор пользователя в `AskUserQuestion` (вариант "Только сам экран, без интеграции").
2. **Email regex `^[^\s@]+@[^\s@]+\.[^\s@]+$`** — простой, согласован с пользователем; перфекционистская валидация не требуется.
3. **`Effect.Submit(email)` несёт email явно** — согласно `MEMORY.md`, в Effector нельзя читать state, данные передаются через поля Effect.

### Риски
Подводные камни, которые могут сорвать реализацию. Не "общие риски проекта", а конкретно для **этой** задачи.

Что писать:
- Технические нюансы (race conditions, edge cases, неочевидное поведение API).
- Зависимости в DI-графе, которые могут уронить компиляцию до полной реализации.
- Известные ограничения тулчейна (Metro Assisted Inject не поддерживает несколько `@Assisted` одного типа; порядок generic'ов в `storeFactory.create<Intent, Action, State, Message, Label>`).
- Что может пойти не так на верификации.

Формат: пронумерованный список. Каждый пункт: **риск** — **митигация / как обойти**.

**Пример:**
1. **AuthGraph уже ссылается на `ForgotPasswordComponentImpl.Factory.bind`** — пока класс не создан, модуль не компилируется. **Митигация:** создавать файлы сверху вниз (Tea → ComponentImpl → Screen), хотя бы скелет ComponentImpl с `@AssistedFactory` нужен раньше остального.
2. **`onFocusChanged` срабатывает и при первом получении фокуса** — наивная проверка `if (!isFocused) validate()` вызовет валидацию до ввода. **Митигация:** флаг `var hadFocus by remember { mutableStateOf(false) }`.

### Следующие шаги
Развёрнутая версия **только невыполненных** пунктов из секции "Текущий план" (тех, что отмечены `⏳`). Готовые пункты (`✅`) сюда НЕ попадают — их детальные инструкции больше не нужны, факт выполнения зафиксирован в плане.

**Соотношение с "Текущим планом":**
- "Текущий план" — однострочный дашборд прогресса со всеми пунктами (`✅`/`⏳`).
- "Следующие шаги" — те же `⏳`-пункты, но **развёрнутые** с конкретикой: пути файлов, code-snippet'ы, критерии завершения, команды верификации.
- При завершении задачи: ставится `✅` в плане, пункт удаляется из "Следующих шагов".
- Нумерация в "Следующих шагах" может быть собственной (1, 2, 3…) — соответствие с планом не требуется.

**Главное правило: каждая задача описывается ЯВНО и САМОДОСТАТОЧНО.** Запрещено отсылать к другим разделам формулировками вида *"реализовать как в плане"*, *"см. таблицу выше"*, *"выполнить пункт 5 из 'Текущий план'"*. Новая сессия должна понять, что делать, читая только секцию "Следующие шаги" — без перекрёстных ссылок.

Первым пунктом всегда — **прочитать STATE_COMPRESSION.md и план** (если есть отдельный plan-файл).

Что писать в каждом шаге:
- **Что именно сделать** — конкретный артефакт (файл с полным путём, метод, gradle-таска).
- **Зависимости** — если шаг N не запустится без шага N-1, указать явно.
- **Критерий завершения** — что должно стать правдой после шага (компиляция прошла, файл создан, тест зелёный).
- **Команда верификации** — конкретная gradle-таска или ручной чек.

Что писать в самой секции:
- Чтение сохранённого контекста (STATE_COMPRESSION.md, план-файл, ключевые образцы кода).
- **Каждая задача реализации — отдельным пунктом с полным путём файла и кратким описанием содержимого** (не "создать MVI" — а "создать `auth-impl/.../mvi/ForgotPasswordTea.kt` с `Tea<State, Intent, Event>`, где State = (email, emailError, isLoading), Intent = (...), Effect = (...), Event = (...)").
- Каждое изменение существующего файла — отдельным пунктом с путём и тем, что именно меняется.
- Команды верификации (компиляция, тесты, smoke).
- Финальная проверка scope: явно перечислить, чего НЕ должно быть в результате (out-of-scope контроль).

Формат: пронумерованный список. Каждый шаг — императив ("Реализовать X", "Запустить Y", "Создать файл Z с содержимым…"). Без отсылок к другим секциям файла.

**Пример** (предположим, в "Текущем плане" пункты 1-3 уже `✅`, а 4-6 ещё `⏳`):

*Текущий план:*
1. ✅ **Domain DTO** — `ForgotPasswordDto.kt`.
2. ✅ **UseCase** — abstract + Impl.
3. ✅ **Request DTO** — `ForgotPasswordRequestDto.kt`.
4. ⏳ **API компонент** — `ForgotPasswordComponent.kt` в `auth-api`.
5. ⏳ **Tea + StoreFactory** — MVI слой.
6. ⏳ **Verification** — `:auth-impl:compileAndroidMain`, `:composeApp:assembleDebug`.

*Следующие шаги (только ⏳, развёрнуто):*
1. Прочитать `STATE_COMPRESSION.md` и план `/path/to/plan.md`.
2. Прочитать образцы: `auth-impl/.../phone_validation/component/PhoneValidationComponentImpl.kt`, `…/PhoneValidationScreen.kt`.
3. Создать `auth-api/.../forgot_password/ForgotPasswordComponent.kt` — интерфейс с `UiState(email, emailError, isLoading)`, методы `onEmailChanged/onSubmitClicked/onBackClicked`, `Callbacks(onSubmitSuccess, onBack)`, `Factory`.
4. Создать `auth-impl/.../mvi/ForgotPasswordTea.kt` — `Tea<State, Intent, Event>` с State, Intent, Effect, Event sealed-типами (привести структуру в подпунктах).
5. Создать `auth-impl/.../mvi/ForgotPasswordStoreFactory.kt` — extension `TeaFactory.forgotPasswordTea(...)` с Reducer и Effector. Email-валидация через regex `^[^\\s@]+@[^\\s@]+\\.[^\\s@]+$`.
6. Запустить `./gradlew :auth-impl:compileAndroidMain` — критерий: 0 unresolved-reference ошибок.
7. Запустить `./gradlew :composeApp:assembleDebug` — критерий: APK собран.
8. Проверить out-of-scope: убедиться, что `PhoneValidationScreen.kt` и `PhoneValidationComponent.kt` НЕ содержат ссылку «Забыли пароль?» и метод `onForgotPasswordClicked`.

**Пример (неправильно — отсылочно, ЗАПРЕЩЕНО):**
1. Прочитать `STATE_COMPRESSION.md`.
2. Реализовать невыполненные пункты из "Текущий план".
3. Запустить компиляцию.

### Поддержание секций в синхроне

При обновлении файла после выполнения работы:
1. В "Текущем плане" заменить `⏳` на `✅` у завершённых пунктов (текст пункта менять не нужно).
2. Из "Следующих шагов" **удалить** соответствующие развёрнутые пункты целиком — они выполнены, детали больше не нужны.
3. Если по ходу работы выявились новые задачи — добавить их и в план (с `⏳`), и в "Следующие шаги" (развёрнуто).
4. Проверить, что число `⏳`-пунктов в плане совпадает с числом задач реализации в "Следующих шагах" (плюс служебные пункты "прочитать контекст", "запустить верификацию").

## Сохранение файла
Сохранить результат в файл `STATE_COMPRESSION.md` в корне проекта.

Если файл уже существует — прочитать его, выявить устаревшие/изменившиеся пункты, обновить только их. Не перезаписывать целиком, чтобы не потерять историю решений.
